# Hardware System Overview

## 1. Purpose

This document describes the current hardware design from a module perspective. It is intended to support later schematic design, system integration, and thesis writing.

The current project stage emphasizes architectural clarity rather than full board completion.

## 2. Module Breakdown

### 2.1 Main Control Module

The main control module is based on an ESP32 development board.

Current responsibilities:

- execute the control algorithm
- read temperature data
- generate PWM output
- provide serial debugging output
- drive the heartbeat status LED

Current validation status:

- validated in Wokwi simulation

### 2.2 Temperature Acquisition Module

The temperature acquisition module is based on the DS18B20 digital temperature sensor.

Current responsibilities:

- provide temperature readings to the controller
- act as the primary temperature sensing interface in the edge node

Current validation status:

- validated in Wokwi simulation
- data line and pull-up logic already represented in the current simulation connection

### 2.3 Actuation Output Module

The actuation output module currently consists of PWM generation from the ESP32.

Current responsibilities:

- output a variable duty-cycle control signal
- represent the control effort generated by the controller

Current validation status:

- validated in Wokwi simulation
- GPIO18 waveform already used for observation

### 2.4 Power Driving Module

The power driving module is currently a design concept rather than a fully implemented circuit.

Planned responsibilities:

- receive the PWM control signal from the ESP32
- switch the heating-side power path through a MOSFET low-side structure
- provide the electrical interface between the low-power controller and the higher-power load

Current validation status:

- not yet implemented as a complete real-hardware circuit
- currently represented only at the design-logic level

### 2.5 Power Module

The power module is currently documented as a staged design direction.

Planned responsibilities:

- accept external input power
- regulate suitable supply rails for the ESP32 and sensor
- provide a common ground reference for sensing and control
- support later load-side power delivery

Current validation status:

- not yet finalized in hardware design
- not yet represented as a complete board-level power solution

### 2.6 Debug and Communication Interface

The current debugging path is based on serial output and status indication.

Current responsibilities:

- provide runtime logs
- support simulation observation
- support later commissioning and test workflows

Current validation status:

- serial logging validated in simulation
- heartbeat LED validated in simulation

## 3. Current Verified vs. Planned Content

### Already verified in simulation

- ESP32 minimum runtime
- DS18B20 temperature acquisition
- GPIO2 LED indication
- GPIO18 PWM output
- serial monitor output
- control-loop behavior based on the virtual thermal model

### Planned for later hardware refinement

- formal power-input design
- voltage-regulation design
- MOSFET power-drive circuit details
- connector definition
- grounding and protection details
- formal schematic and later PCB implementation

## 4. Current Engineering Position

At the present stage, the hardware work should be understood as a staged abstraction built on top of simulation results.

The simulation has already verified the logical relationships between controller, sensing, and actuation. The next hardware-design tasks are therefore not starting from zero, but translating that validated logic into a clearer and more realistic electrical design.
